CASE STUDY / 02

Smart Dairy

End-to-end dairy farm management with lactation modeling and estrus detection.

YEAR
2025
CLIENT

ROLE

STATUS
Live · In production
§ 01

The problem

Modern dairy farming demands precise tracking of each cow's reproductive cycle, milk yield, health events, and feeding regimen. Most Kenyan dairy operations still rely on paper records, which makes it nearly impossible to spot reproductive issues early, optimize feed efficiency, or model lactation curves to predict yield.

Kapgrade needed a system that treated the individual cow as the central data entity, not the farm or the herd. Every event in a cow's life needed to be tied to that animal, queryable across time, and aggregable into farm-level KPIs. Specifically, it needed to:

›Maintain individual cow profiles with full lineage and complete event history
›Detect estrus cycles in real time and alert the farm team for breeding windows
›Model each cow's lactation curve and predict daily yield deviations
›Log milk production twice daily (AM and PM) across the entire herd
›Track health events, breeding events, and feed costs at the cow level
›Surface herd-level KPIs derived from the underlying per-cow data
§ 02

The approach

The data model puts each cow at the center. Around each cow sit four event types: reproductive events, health events, milk production records, and feed records. Every other view in the application (the herd dashboard, the morning report, the financial summary) is derived from this base.

The interface follows the same logic: the cow's profile page is the primary work surface, with all events for that animal visible on a single timeline. Frontend is Next.js + TypeScript; backend is Flask + Python; database is Supabase-managed PostgreSQL with domain constraints enforced at the database layer rather than the application layer.

§ 03

Key features

›Individual cow profiles with full lineage and complete health history
›Estrus detection alerts based on cycle modeling and time since last calving
›Lactation curve fitting per cow, with daily yield prediction and deviation alerts
›AM/PM milk production logging with a single-tap entry flow
›Health event tracking: calving, illness, vaccination, artificial insemination, deworming
›Feed log with cost-per-liter analysis at both the cow and herd level
›Herd-level dashboard: average yield, conception rate, calving interval
§ 04

Technical highlights

›Lactation curve modeling using Wood's incomplete gamma function, Y(t) = a · t^b · e^(-ct), fitted per cow per lactation
›Estrus cycle prediction with deviation tracking when cycles run shorter or longer than the standard 21-day baseline
›PostgreSQL CHECK constraints enforce domain rules at the database layer; for example, no AI event recordable within 50 days post-calving
›Real-time charts via Recharts for yield trends, herd KPIs, and individual lactation curves
›Soft deletes everywhere so historical data is preserved even when a cow leaves the herd
›Server Components by default for fast page loads on rural network conditions
§ 05

Challenges & solutions

Modeling biology with code.

A cow's lactation curve is not a straight line. It rises sharply post-calving, peaks around six to eight weeks, then declines slowly. Standard linear regression doesn't fit. I used Wood's gamma function with parameters fitted per cow per lactation, which lets the system both predict next-week yield and flag cows whose actual production is deviating from their personal curve.

Reliable data entry by farm workers.

The system has to be usable by farm staff entering data each morning, often on a phone, sometimes with cold hands. I built the milk log as a single tap-per-cow flow: select cow, enter AM yield, submit. No multi-step forms, no required fields beyond the yield itself. The friction-to-record gap is what kills paper-replacement projects in agricultural settings.

Enforcing domain rules across teams.

Different workers handle different events: the milker logs production, the herdsman logs breeding, the vet logs health. Mistakes happen. I moved domain rules (no AI before 50 days post-calving, no calving without a prior pregnancy event, no milk record without an active lactation) into PostgreSQL CHECK constraints. The application layer never has to remember to validate, because the database refuses bad data outright.

§ 07

Lessons learned

→Domain rules belong in the database, not the application layer. They survive every refactor and every new developer.
→When designing for workers in the field, every extra tap is a reason the system gets abandoned. Default to single-action flows.
→Mathematical models built for biology need cow-specific parameters, not herd averages. The whole point of the system is to see individuals.
