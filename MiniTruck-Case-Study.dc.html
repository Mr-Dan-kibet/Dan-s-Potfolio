MiniTruck

Uber for the school run: book a seat on a bus already passing your home-to-school route, any school, any number of days.

§ 01

The problem

The school run is a daily logistics headache for Kenyan families. Parents either drive their own children (burning hours and fuel on routes other people are already driving) or commit to a single school's bus contract that is rigid, expensive, and tied to one institution. Children attending different schools in the same neighborhood have no shared option.

There was no way to do the obvious thing: put a child on a bus that already passes the house on its way past their school, and pay only for the days actually needed. MiniTruck set out to make that the default. It needed to:

›Match a child's home and school to bus routes that already pass both points
›Let parents book by the day, week, or term, not a rigid annual contract
›Work across schools, so a route serves whichever children live along it
›Give drivers a manifest and parents visibility into who is on which bus
§ 02

The approach

MiniTruck is modeled as a two-sided marketplace: routes on one side, bookings on the other. A route is an ordered set of geographic stops with a schedule and a capacity; a booking attaches a child to a route for a chosen set of days. The matching layer sits between them, surfacing only the routes that pass near both a child's home and their school.

The build is a React frontend talking to a Flask + Python REST API over a PostgreSQL database. As my Moringa capstone, the goal was to demonstrate end-to-end ownership: data model, API design, auth, booking logic, and a deployed, working product, not just a UI mockup.

§ 03

Key features

›Route matching by home + school location, independent of which school the child attends
›Flexible booking, per day, per week, or per term, with pricing that follows the selection
›Separate parent and driver roles, each with their own dashboard
›Per-route capacity tracking so a bus can't be overbooked
›Driver manifest showing which children board at which stop on a given day
›Parent view of upcoming trips and booking history per child
§ 04

Technical highlights

›Route-to-booking data model where a single route serves many children across many schools
›Day-level booking model where a booking is a set of dates, not a single contract, so pricing and capacity compute per day
›Role-based access separating parent and driver capabilities behind a shared auth layer
›Flask REST API with a normalized PostgreSQL schema for routes, stops, bookings, and users
›Deployed end to end, taken from architecture through a live, working build for capstone review
§ 05

§ 06

Lessons learned

→Modeling a booking as a set of days instead of a single contract unlocked the whole flexible-pricing idea. The data model decides what the product can do.
→Two-sided marketplaces are mostly a matching problem. Get the route↔booking relationship right and the dashboards fall out of it.
→Shipping a deployed product for a capstone, not a slide deck, is what made the learning real and the work portfolio-worthy.
